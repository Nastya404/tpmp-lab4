Описание задачи

Информационная система предназначена для автоматизации работы автопарка. Она позволяет вести учёт транспортных средств, анкетных данных водителей и истории выполненных заказов. Основные функции:

  • Контроль грузоподъёмности при назначении заказов.
  • Расчёт заработной платы водителей (20% от стоимости поездки).
  • Генерация статистических отчётов по пробегу и эффективности персонала.
  • Ролевая модель доступа: диспетчер (ADMIN) и водитель (DRIVER).

Текстовые сценарии

Добавление заказа UC-01
Краткое описание
Позволяет диспетчеру зарегистрировать новую перевозку.
Область действия
ИС «Автопарк»
Уровень
Цель пользователя
Предусловия
Диспетчер авторизован; водитель и автомобиль созданы в БД.
Основной поток
1. Диспетчер инициирует добавление заказа через меню.
2. Система запрашивает параметры (Дата, ID водителя, № авто, Масса, Стоимость).
3. Диспетчер вводит данные.
4. Система проверяет наличие объектов в БД.
5. Система выполняет проверку грузоподъёмности (include).
6. Система подтверждает корректность веса груза.
7. Система сохраняет запись и обновляет пробег авто.
8. Система уведомляет об успехе.
Альт. потоки
5А. Перегруз: Система сообщает, что масса груза превышает грузоподъёмность, и отклоняет сохранение.
Постусловия
Данные о заказе сохранены; водителю начислено 20% от стоимости.
Расчёт зарплаты водителя за период UC-02
Краткое описание
Позволяет диспетчеру рассчитать суммарное вознаграждение конкретного водителя за выбранный период.
Предусловия
Диспетчер авторизован; есть выполненные заказы для данного водителя.
Основной поток
1. Диспетчер выбирает функцию «Расчёт ЗП конкретного водителя».
2. Система запрашивает фамилию водителя и даты начала/конца периода.
3. Система делает выборку всех заказов водителя из таблицы ORDERS.
4. Система суммирует стоимость поездок и вычисляет 20%.
5. Результат выводится на экран и заносится в SALARY_RECORDS.
Альт. потоки
4А. Водитель не найден: Система сообщает об ошибке.
5А. Нет заказов: Система выводит «Сумма: 0».
Формирование статистики по машине UC-03
Краткое описание
Вывод общей массы перевезённых грузов и суммарного пробега для конкретного автомобиля.
Основной поток
1. Диспетчер выбирает «Статистика по машине».
2. Система запрашивает государственный номер автомобиля.
3. Система суммирует километраж и прибавляет к начальному пробегу.
4. Система отображает полные сведения об авто, общий пробег и массу.
Альт. потоки
4А. Машина не найдена: «Транспортное средство с таким номером не зарегистрировано».
Просмотр водителем личных данных UC-04
Краткое описание
Позволяет водителю получить информацию о своей машине и своих выполненных заказах.
Предусловия
Водитель авторизован в системе под своим табельным номером.
Основной поток
1. Водитель выбирает «Просмотр информации о закреплённой машине».
2. Система идентифицирует машину по табельному номеру.
3. Водитель вводит период и получает отфильтрованный список заказов.
4. Система показывает начисленную зарплату.
Постусловия
Водитель не может просматривать данные других сотрудников.
Дополнительная спецификация
Нефункциональные требования

Документ определяет нефункциональные требования к информационной системе «Автопарк».

1. Надёжность (Reliability)

  • Целостность данных: Система обеспечивает корректность связей между таблицами CARS, DRIVERS и ORDERS с использованием FOREIGN KEY и ON UPDATE CASCADE.
  • Валидация при вводе: Триггеры БД (trg_orders_cargo_mass_ins, trg_orders_cargo_mass_upd) предотвращают сохранение заказов с превышением грузоподъёмности.
  • Отказоустойчивость: При ошибках в SQL-запросах система выводит информативное сообщение, не завершая работу аварийно.

2. Производительность (Performance)

  • Скорость отклика: Время выполнения расчётных функций не превышает 1–2 секунд на стандартной рабочей станции.
  • Эффективность БД: Primary Keys в SQLite обеспечивают мгновенный поиск по номеру машины или табельному номеру водителя.

3. Удобство использования (Usability)

  • Интерфейс: Консольное меню (CLI) с чётким разделением прав для ролей ADMIN и DRIVER.
  • Локализация: Весь интерфейс и системные сообщения на русском языке.
  • Обучаемость: Не более 10 минут на освоение базовых операций диспетчера.

4. Сопровождаемость (Supportability)

  • Модульность: Структура каталогов bin/, build/, include/, src/; разделение кода на логические модули (db_core, crud, business_logic, reports).
  • Документированность: Комментарии к ключевым структурам данных (Models) и функциям API.
  • Расширяемость: Архитектура позволяет добавлять новые типы отчётов без изменения структуры существующих таблиц.

Описание таблиц

ТаблицаОписаниеКлючевые поля
AUTOPARK_CONFIGГлобальные настройки системы (ключ-значение).config_key PK
AUTOPARK_USERSУчётные записи: логины, хэши паролей, роли ADMIN/DRIVER.user_id PK, login UNIQUE
CARSСписок транспортных средств, пробег и грузоподъёмность.number PK
DRIVERSАнкетные данные водителей, стаж и категории прав.personnel_number PK
ORDERSЖурнал выполненных рейсов: водитель, машина, груз.order_id PK, FK на DRIVERS и CARS
SALARY_RECORDSАрхив начисленных выплат водителям за периоды.record_id PK, UNIQUE (driver_id, start_date, end_date)

Ключевые ограничения

  • Все связи реализованы через FOREIGN KEY с ON UPDATE CASCADE.
  • Триггер trg_orders_cargo_mass_ins блокирует заказ при превышении грузоподъёмности.
  • Триггер trg_orders_cargo_mass_upd применяется при обновлении заказа.
  • Поле role в AUTOPARK_USERS принимает только ADMIN или DRIVER (CHECK constraint).
  • Поле category в DRIVERS: допустимые значения A, B, C, D, E, CE, DE.
  • Уникальность в SALARY_RECORDS по (driver_id, start_date, end_date) — исключает дубли при пересчёте.
